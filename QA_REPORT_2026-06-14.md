# Sharia Scope Analyzer QA Report

Date: 14 June 2026

## Executive assessment

The rebuilt product is now an analyzer, not a lookup tool. It accepts raw financial line items, computes the PSX/KMI ratios, produces a verdict and reasons, calculates dividend purification, validates the formula engine against a Meezan-labelled dataset, and exports a PDF tear-sheet.

The formula engine is a credible V1 foundation. The app is not yet reliable enough for unsupervised financial-statement extraction or final user release. The main remaining risks are extraction traceability, stale results after input edits, cost-heavy defaults, unit ambiguity, and report layout.

## Real-document cost test

I used a reduced 13-page financial-statement section from the supplied 43-page Lucky Cement Q3 FY24 report and AWS Bedrock Claude Haiku 4.5.

- Input tokens: 30,859
- Output tokens: 804
- Runtime: 10.52 seconds
- Estimated model cost at published Haiku 4.5 rates: USD 0.034879
- Result: structured extraction completed, but market price was absent and the verdict correctly remained Review Required.

The extraction was not fully trustworthy: Haiku treated income from deposits with Islamic banks as non-compliant and produced materially different values from the earlier Opus run. This demonstrates that model output must be evidence-backed and confirmed by the user before screening.

A second Bedrock call failed because the AWS account had not completed model use-case access. The current UI's credential indicator can therefore say extraction is enabled even when the selected model cannot be invoked.

## Findings

### P1: The shared Anthropic key must be rotated

The live Anthropic key was pasted into conversation history. A repository scan found no copy of it in the project files, but chat exposure is still sufficient reason to revoke it.

Recommendation: delete/revoke that key in the Anthropic Console, create a replacement, and keep the replacement in an environment variable or local secret store rather than chat or source code.

### P1: Extracted classifications have no source evidence

The AI is asked to determine interest-bearing debt, non-compliant investments, non-compliant income, and illiquid assets. These are not simple transcription fields; they require Shariah and accounting classification. The response stores only values and a free-text note, with no page number, source line, calculation, or confidence per field.

Impact: a plausible but wrong classification can silently change the verdict or purification amount.

Recommendation: return evidence for every extracted field: source page, quoted label, raw amount, inclusion/exclusion rationale, and confidence. Require explicit confirmation for classification-heavy fields before enabling the final verdict.

### P1: Results become stale after inputs are edited

After screening a compliant case, changing debt from 10% to 50% without pressing Run left the green Compliant badge and old PDF available. The result tuple is stored in session state and is not invalidated when form values change.

Impact: the screen can display a verdict that no longer matches the visible inputs.

Recommendation: recompute automatically on input change, or store an input fingerprint and hide results/report with a clear "Inputs changed - run screening again" state.

### P1: Share-count units can make NLA per share incorrect

The UI says financial amounts should use one unit, such as PKR '000, but does not explain that the share count must use a corresponding scale. PKR '000 divided by an absolute share count is 1,000 times too small. AI extraction can also return shares in a different scale from statement amounts.

Impact: the net-liquid-assets-per-share screen can be wrong while all entered numbers appear reasonable.

Recommendation: capture currency scale and share-count scale separately, normalize both to base units, and show the exact NLA calculation before screening.

### P1: Default AI configuration is expensive and partly invalid

The Anthropic selector defaults to Opus 4.8. The Bedrock default model ID is also Opus-based and did not match the accessible cross-region model profile in the tested AWS account. The region defaults to us-east-1 despite the configured local AWS region being eu-west-1.

Impact: unnecessary cost and failed first-run extraction.

Recommendation: default to Haiku 4.5, use the AWS default credential chain and configured region, verify model access before showing enabled, and display estimated/actual tokens and cost for each extraction. Offer Sonnet as the deliberate fallback when confidence is low.

### P2: Full PDF upload is token-inefficient

The app sends the complete PDF as a visual document. It has no local page selection, text-layer inspection, repeated-header removal, or token estimate. Even the reduced 13-page document used 30,859 input tokens.

Recommendation: locally detect/select standalone balance sheet, income statement, cash/investment notes, borrowings notes, and share-capital pages. Let the user review the selected pages before the API call.

### P2: Validation coverage and explanation are incomplete

The parsed validation file contains 503 rows rather than all 537 reported index rows. Of 12 mismatches, four have no note in the generated CSV: HUBC, ITANZ, NETSOL, and TELE. Therefore the mismatches are not all proven manual exceptions.

Recommendation: show coverage as 503/537, separate documented exceptions from unexplained mismatches, and manually reconcile the missing 34 rows and four unnoted disagreements.

### P2: Market price is a disconnected required input

Market price usually does not exist in financial statements. Extraction therefore often ends in Review Required even when statement extraction succeeds.

Recommendation: after extraction, present a dedicated completion step for market price, price date, and source. Do not imply that the PDF alone can produce a final verdict.

### P2: Exported PDF has a visible layout collision

In the Lucky Cement tear-sheet, the long review-period value overlaps the Source label. Metadata columns have fixed widths and do not adapt to longer content.

Recommendation: use a two-row stacked metadata layout or wrap values in Paragraph objects with tested column widths. Add a rendered-PDF regression test.

### P2: Report text is interpreted as ReportLab markup

An input such as `Alpha <b broken` causes PDF generation to fail, while valid markup can alter report styling.

Recommendation: XML-escape all user/AI-provided text before passing it to ReportLab Paragraph.

### P3: AWS credential UX encourages manual secret entry

The sidebar prominently asks for access and secret keys even though the SDK supports the standard AWS credential chain.

Recommendation: default to profile/environment/IAM credentials, add an optional advanced credentials section, and never encourage long-lived keys in the normal workflow.

### P3: Automated coverage is narrower than the product surface

The 17 tests cover core calculations, but there are no committed tests for AI provider configuration, structured extraction parsing, Streamlit state invalidation, PDF rendering, special-character handling, or end-to-end report download.

## UI/UX assessment

Strengths:

- The Analyze / Validation / Methodology structure is understandable.
- Dark mode is readable.
- The form and ratio cards adapt to mobile without horizontal overflow.
- Status colors and failure reasons are clear.
- Validation visibly establishes that the index is an oracle, not the product data source.

Improvements:

- Turn Analyze into a visible three-step workflow: Document and period, Extract and verify, Screen and export.
- Keep AI configuration inside the Analyze flow; it consumes permanent sidebar space on Validation and Methodology.
- Replace decorative feature pills with review status: document selected, extraction complete, fields verified, price supplied.
- Add field-level evidence drawers and warnings for inferred values.
- Mark unscreened, stale, and verified states distinctly.
- Tighten the mobile tab labels; the emoji-led labels are crowded at 390 px.

## Recommended release order

1. Fix stale-result invalidation and NLA unit normalization.
2. Add evidence/page references and confirmation for every AI-extracted field.
3. Default to Haiku 4.5 and add page selection plus token/cost reporting.
4. Add the market-price completion step.
5. Fix PDF wrapping/escaping and add render tests.
6. Reconcile validation coverage and unexplained mismatches.
7. Polish the Analyze workflow and provider setup.

## Decision

Good analyzer prototype: yes.

Ready for supervised demos: yes, if every extracted value is manually checked.

Ready for users to upload statements and trust the verdict without review: no.
